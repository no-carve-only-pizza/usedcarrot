# UsedCarrot 시스템 설계

## 1. 설계 개요

UsedCarrot은 중고 상품을 등록하고, 상품을 조회하며, 사용자 간 채팅과 CarrotCoin 모의지갑 거래를 통해 거래를 진행할 수 있는 웹 기반 중고거래 플랫폼이다. 본 문서는 SRS에서 정의한 요구사항을 실제 화면, 기능, 데이터, 보안 규칙 단위로 구체화한다.

## 2. 전체 시스템 구성

- 프론트엔드
  - 사용자가 웹 브라우저에서 접근하는 화면을 제공한다.
  - 로그인, 회원가입, 상품 목록, 상품 상세, 상품 등록, 채팅, 마이페이지, 관리자 화면을 포함한다.
  - 서버 API 호출 시 인증 토큰 또는 세션 정보를 함께 전달한다.

- 백엔드 서버
  - 사용자 요청을 처리하는 API를 제공한다.
  - 인증, 권한 확인, 입력값 검증, 비즈니스 로직, 보안 로그 기록을 담당한다.
  - 클라이언트에서 전달한 사용자 ID를 신뢰하지 않고 서버의 인증 정보를 기준으로 처리한다.

- 데이터베이스
  - 사용자, 상품, 이미지, 관심 상품, 채팅 메시지, 모의지갑, CarrotCoin 거래 내역, 신고, 감사 로그를 저장한다.
  - 비밀번호는 해시값으로 저장한다.
  - 삭제 또는 정지 상태는 별도 상태값으로 관리한다.

- 파일 저장소
  - 상품 이미지를 저장한다.
  - 업로드 파일명은 서버에서 새로 생성한다.
  - 실행 가능한 파일이 저장되거나 실행되지 않도록 분리된 위치에 보관한다.

## 3. 사용자 가입 및 계정 기능

- 사람들이 플랫폼에 가입할 수 있어야 한다.
  - 회원가입 페이지가 있어야 한다.
  - 회원가입 페이지에는 이메일, 비밀번호, 비밀번호 확인, 닉네임, 지역 입력란이 있어야 한다.
  - 이메일은 로그인 ID로 사용한다.
  - 닉네임은 상품, 채팅, 프로필 화면에서 표시된다.
  - 이메일과 닉네임은 중복될 수 없다.
  - 비밀번호는 최소 길이와 복잡도 정책을 만족해야 한다.
  - 회원가입 성공 시 사용자는 로그인 페이지 또는 홈 화면으로 이동한다.

- 사람들이 플랫폼에 로그인할 수 있어야 한다.
  - 로그인 페이지가 있어야 한다.
  - 사용자는 이메일과 비밀번호로 로그인한다.
  - 로그인 성공 시 서버는 세션 또는 인증 토큰을 발급한다.
  - 로그인 실패 시 "이메일 또는 비밀번호가 올바르지 않습니다"와 같은 일반 메시지를 보여준다.
  - 이메일 존재 여부나 비밀번호 오류 여부를 구체적으로 알려주지 않는다.
  - 일정 횟수 이상 실패하면 일시적으로 로그인을 제한할 수 있다.

- 사용자는 자신의 프로필을 확인할 수 있어야 한다.
  - 프로필 페이지에서 닉네임, 지역, 소개글, 가입일, 거래 관련 정보를 볼 수 있다.
  - 다른 사용자의 프로필에서는 이메일 같은 민감 정보가 보이지 않아야 한다.
  - 본인 프로필에서는 소개글, 지역, 비밀번호를 수정할 수 있어야 한다.
  - 비밀번호 변경 시 현재 비밀번호를 다시 확인해야 한다.

- 마이페이지 기능이 필요하다.
  - 내가 등록한 상품 목록을 볼 수 있어야 한다.
  - 내가 관심 등록한 상품 목록을 볼 수 있어야 한다.
  - 내가 참여한 채팅 목록을 볼 수 있어야 한다.
  - 내 CarrotCoin 지갑 잔액과 거래 내역을 볼 수 있어야 한다.
  - 계정 탈퇴 기능을 제공해야 한다.

- 유저 정보는 데이터베이스로 관리해야 한다.
  - 이메일, 닉네임, 비밀번호 해시, 지역, 소개글, 권한, 상태를 저장한다.
  - 비밀번호 원문은 저장하지 않는다.
  - 정지된 계정은 로그인, 상품 등록, 채팅, CarrotCoin 구매, 신고 기능을 사용할 수 없다.

## 4. 상품 등록 및 조회 기능

- 사용자들은 상품을 올리고 볼 수 있어야 한다.
  - 상품 등록 페이지가 있어야 한다.
  - 로그인한 사용자만 상품을 등록할 수 있다.
  - 상품 등록 시 상품명, 가격, 카테고리, 설명, 거래 희망 지역, 상품 이미지를 입력한다.
  - 상품명, 가격, 설명, 카테고리는 필수값이다.
  - 가격은 0원 이상의 숫자만 허용한다.
  - 상품 설명은 최대 글자 수를 제한한다.

- 상품 이미지를 등록할 수 있어야 한다.
  - 상품 이미지는 여러 장 등록할 수 있다.
  - 허용 확장자는 jpg, jpeg, png, webp 등 이미지 파일로 제한한다.
  - 파일 크기는 정해진 최대 용량을 넘을 수 없다.
  - 서버는 확장자, MIME 타입, 파일 크기를 검증한다.
  - 저장 파일명은 사용자가 올린 이름을 그대로 쓰지 않고 UUID 등으로 새로 만든다.
  - 업로드 실패 시 어떤 파일이 실패했는지 사용자에게 알려준다.

- 등록된 상품은 누구나 볼 수 있어야 한다.
  - 홈 화면에는 최신 상품 목록을 보여준다.
  - 상품 목록에서는 상품 이미지, 상품명, 가격, 지역, 등록 시간을 보여준다.
  - 목록에서는 설명 전체를 보여주지 않고 요약 정보만 보여준다.
  - 상품을 클릭하면 상세 페이지로 이동한다.
  - 상세 페이지에서는 상품명, 가격, 설명, 이미지, 판매자 정보, 거래 상태, 등록일을 보여준다.

- 판매자는 자신이 등록한 상품을 관리할 수 있어야 한다.
  - 내가 등록한 상품 목록 페이지가 있어야 한다.
  - 판매자는 본인 상품만 수정할 수 있다.
  - 판매자는 본인 상품만 삭제할 수 있다.
  - 판매자는 상품 상태를 판매중, 예약중, 거래완료로 변경할 수 있다.
  - 삭제된 상품은 일반 사용자 목록과 검색 결과에 보이지 않아야 한다.

- 상품 정보는 데이터베이스로 관리해야 한다.
  - 상품 테이블에는 판매자 ID, 상품명, 가격, 설명, 카테고리, 지역, 거래 상태, 조회수, 생성일, 수정일을 저장한다.
  - 상품 이미지 테이블에는 상품 ID, 저장 파일명, 원본 파일명, MIME 타입, 파일 크기, 저장 경로를 저장한다.
  - 상품과 이미지는 1:N 관계로 관리한다.

## 5. 상품 검색 및 필터 기능

- 사용자는 상품을 검색할 수 있어야 한다.
  - 홈 화면 또는 검색 페이지에 검색창이 있어야 한다.
  - 사용자는 키워드로 상품명과 설명을 검색할 수 있다.
  - 검색어는 최대 길이를 제한한다.
  - 검색어에 특수문자가 포함되어도 SQL 조작이 발생하지 않아야 한다.

- 사용자는 상품을 조건별로 볼 수 있어야 한다.
  - 카테고리별 필터가 있어야 한다.
  - 거래 상태별 필터가 있어야 한다.
  - 지역별 필터를 제공할 수 있다.
  - 가격 범위 필터를 제공할 수 있다.
  - 검색 결과는 최신순을 기본으로 정렬한다.

- 검색 결과는 서버 부하를 고려해야 한다.
  - 한 번에 모든 상품을 내려주지 않는다.
  - 페이지네이션 또는 무한 스크롤을 사용한다.
  - 검색 결과가 없을 때는 빈 상태 화면을 보여준다.

## 6. 사용자 간 소통 기능

- 플랫폼 사용자들끼리 소통이 가능해야 한다.
  - 상품 상세 페이지에 판매자에게 문의하기 버튼이 있어야 한다.
  - 로그인한 사용자만 채팅 또는 메시지를 보낼 수 있다.
  - 구매자는 상품 판매자에게 메시지를 보낼 수 있어야 한다.
  - 판매자는 구매자에게 답장할 수 있어야 한다.

- 1:1 채팅이 가능해야 한다.
  - 각 상품마다 구매자와 판매자 사이의 1:1 채팅방이 생성된다.
  - 같은 구매자가 같은 상품에 다시 문의하면 기존 채팅방을 재사용한다.
  - 채팅방 목록에서는 상대방 닉네임, 상품명, 마지막 메시지, 마지막 메시지 시간을 보여준다.
  - 채팅방 상세에서는 메시지 내용과 보낸 시간을 보여준다.
  - 채팅 메시지는 시간순으로 정렬한다.

- 전체 공지 또는 공개 소통 기능은 선택 기능으로 둔다.
  - 수업 범위가 충분하면 전체 사용자에게 공지하는 게시판을 만들 수 있다.
  - 전체 채팅은 악용 위험이 크므로 1차 구현 범위에서는 제외할 수 있다.

- 채팅 정보는 데이터베이스로 관리해야 한다.
  - 채팅방에는 상품 ID, 구매자 ID, 판매자 ID, 생성일을 저장한다.
  - 메시지에는 채팅방 ID, 보낸 사람 ID, 메시지 내용, 생성일, 읽음 여부를 저장한다.
  - 채팅방 참여자와 관리자만 메시지를 조회할 수 있다.

- 채팅에는 보안 처리가 필요하다.
  - 메시지 내용은 최대 길이를 제한한다.
  - 메시지에 스크립트를 입력해도 화면에서 실행되지 않아야 한다.
  - 정지된 사용자는 메시지를 보낼 수 없다.
  - 삭제되거나 거래완료된 상품에는 신규 문의를 제한할 수 있다.

## 7. CarrotCoin 모의지갑 및 거래 상태 기능

- 유저들은 CarrotCoin 모의지갑으로 상품을 구매할 수 있어야 한다.
  - 실제 블록체인, 실제 코인, 개인키, 외부 결제 API는 사용하지 않는다.
  - 회원가입 시 사용자별 지갑을 자동 생성한다.
  - 신규 사용자의 초기 잔액은 0 CarrotCoin으로 설정하고, 관리자가 제한된 금액을 지급한다.
  - 상품 상세 페이지 또는 채팅방에서 CarrotCoin 구매 버튼을 제공한다.
  - 구매자는 판매중 또는 예약중 상품에 대해서만 CarrotCoin 구매를 할 수 있다.
  - 구매자와 판매자가 같은 경우 구매 요청을 만들 수 없다.
  - 거래 내역에는 상품 ID, 구매자 ID, 판매자 ID, 금액, 거래 유형, 상태, transactionHash, idempotencyKey, 생성일을 저장한다.

- CarrotCoin 구매는 지갑 잔액으로 처리해야 한다.
  - 구매 금액은 클라이언트에서 받지 않고 서버의 상품 가격을 사용한다.
  - 구매자 잔액이 부족하면 구매를 거부한다.
  - 구매 성공 시 구매자 잔액은 감소하고 판매자 잔액은 증가한다.
  - 구매 성공 후 상품 상태는 거래완료로 변경된다.
  - 완료된 거래 상태는 일반 사용자가 임의로 실패나 취소로 바꿀 수 없다.

- CarrotCoin 거래 중복 처리를 막아야 한다.
  - 구매 요청에는 멱등성 키를 사용한다.
  - 같은 요청이 여러 번 제출되어도 한 번만 처리한다.
  - 사용자가 버튼을 여러 번 눌러도 잔액이 중복 차감되지 않아야 한다.

- 지갑과 거래 내역을 확인할 수 있어야 한다.
  - 사용자는 본인 지갑 잔액을 볼 수 있다.
  - 구매자는 본인이 구매한 CarrotCoin 거래 내역을 볼 수 있다.
  - 판매자는 본인 상품 판매로 입금된 CarrotCoin 거래 내역을 볼 수 있다.
  - 관리자는 분쟁 확인을 위해 거래 상태와 감사 로그를 볼 수 있다.
  - 타인의 지갑 잔액과 거래 내역은 조회할 수 없다.

- CarrotCoin 모의지갑에는 보안 처리가 필요하다.
  - 지갑 잔액은 음수가 될 수 없다.
  - 구매자 잔액 차감과 판매자 잔액 증가는 하나의 DB 트랜잭션으로 처리한다.
  - 실제 가상자산 개인키, 지갑 주소, 외부 API 키를 저장하지 않는다.
  - 거래 해시는 UUID로 생성한 모의 transactionHash를 사용한다.

## 8. 신고 및 차단 기능

- 악성 유저나 상품을 차단해야 한다.
  - 상품 상세 페이지에 신고 버튼이 있어야 한다.
  - 사용자 프로필 또는 채팅방에 사용자 신고 버튼이 있어야 한다.
  - 신고 시 신고 사유를 선택하거나 직접 작성할 수 있어야 한다.
  - 신고 사유에는 사기 의심, 금지 물품, 욕설/비방, 허위 정보, 기타가 포함될 수 있다.

- 신고 정보는 데이터베이스로 관리해야 한다.
  - 신고자 ID, 신고 대상 유형, 신고 대상 ID, 신고 사유, 처리 상태, 생성일을 저장한다.
  - 같은 사용자가 같은 대상을 반복 신고하는 경우 중복 신고를 제한할 수 있다.
  - 신고 내용에는 스크립트가 실행되지 않도록 처리한다.

- 일정 횟수 이상 신고된 상품은 차단되어야 한다.
  - 신고가 일정 기준 이상 누적된 상품은 자동으로 숨김 상태로 변경할 수 있다.
  - 숨김 처리된 상품은 일반 목록과 검색 결과에 보이지 않는다.
  - 관리자는 숨김 처리된 상품을 검토하고 복구 또는 삭제할 수 있다.

- 일정 횟수 이상 신고된 유저는 휴면 또는 정지 계정으로 전환할 수 있다.
  - 신고가 일정 기준 이상 누적된 사용자는 자동으로 제한 상태가 될 수 있다.
  - 제한 상태 사용자는 상품 등록, 메시지 전송, CarrotCoin 구매를 할 수 없다.
  - 관리자는 사용자 상태를 정상, 제한, 정지로 변경할 수 있다.

## 9. 관리자 기능

- 관리자가 플랫폼의 모든 요소를 관리할 수 있어야 한다.
  - 관리자 로그인 또는 관리자 권한 확인이 필요하다.
  - 관리자 화면은 일반 사용자에게 보이지 않아야 한다.
  - 관리자 API는 관리자 권한을 가진 사용자만 호출할 수 있어야 한다.

- 관리자는 사용자를 관리할 수 있어야 한다.
  - 사용자 목록을 조회한다.
  - 사용자 상세 정보를 확인한다.
  - 사용자의 상태를 정상, 제한, 정지로 변경한다.
  - 단, 비밀번호 원문은 어떤 경우에도 볼 수 없다.

- 관리자는 상품을 관리할 수 있어야 한다.
  - 전체 상품 목록을 조회한다.
  - 신고된 상품을 우선 확인한다.
  - 부적절한 상품을 숨김 또는 삭제 처리한다.
  - 숨김 처리 사유를 기록한다.

- 관리자는 신고를 처리할 수 있어야 한다.
  - 신고 목록을 조회한다.
  - 신고 상세 내용을 확인한다.
  - 신고 상태를 접수, 처리중, 처리완료, 반려로 변경한다.
  - 처리 결과와 처리자를 기록한다.

- 관리자는 감사 로그를 확인할 수 있어야 한다.
  - 로그인 실패, 권한 오류, 신고 처리, 사용자 정지, 상품 삭제, CarrotCoin 거래 성공/실패 기록을 볼 수 있다.
  - 로그에는 비밀번호, 세션 ID, 인증 토큰, 실제 가상자산 개인키, 외부 API 키가 포함되지 않아야 한다.

## 13. 보안 설계

- 인증 보안
  - 비밀번호는 안전한 해시 알고리즘으로 저장한다.
  - 로그인 실패 메시지는 일반화한다.
  - 인증 토큰 또는 세션은 예측하기 어렵게 생성한다.
  - 로그아웃 시 세션 또는 토큰을 무효화한다.

- 인가 보안
  - 본인 상품만 수정 또는 삭제할 수 있다.
  - 채팅 참여자만 채팅 내용을 볼 수 있다.
  - 지갑 소유자만 지갑 잔액을 볼 수 있다.
  - CarrotCoin 거래 참여자만 거래 내역을 볼 수 있다.
  - 관리자 API는 관리자만 접근할 수 있다.

- 입력값 검증
  - 모든 요청 데이터는 서버에서 검증한다.
  - 문자열 길이를 제한한다.
  - 가격, ID, 상태값은 허용된 형식만 받는다.
  - HTML 또는 스크립트 입력은 화면 출력 시 실행되지 않도록 처리한다.

- 파일 업로드 보안
  - 이미지 파일만 업로드할 수 있다.
  - 확장자, MIME 타입, 파일 크기를 검증한다.
  - 저장 파일명은 서버에서 생성한다.
  - 업로드 폴더에는 실행 권한을 부여하지 않는다.

- 로그 보안
  - 로그인 실패, 권한 오류, 신고 처리, 관리자 조치, CarrotCoin 거래 성공/실패를 기록한다.
  - 비밀번호, 인증 토큰, 세션 ID, 실제 가상자산 개인키, 외부 API 키는 로그에 저장하지 않는다.
  - 반복 실패 요청이나 의심스러운 요청은 감사 로그에서 확인할 수 있어야 한다.

## 14. 비기능 설계

- 보안
  - HTTPS 사용을 전제로 한다.
  - 인증과 권한 검사는 서버에서 수행한다.
  - 민감 정보는 최소한으로 수집하고 노출하지 않는다.

- 디자인
  - 홈, 상품 상세, 상품 등록, 채팅, 마이페이지는 모바일에서도 사용할 수 있어야 한다.
  - 중요한 버튼은 명확하게 구분한다.
  - 삭제, 신고, CarrotCoin 구매, 계정 탈퇴처럼 중요한 작업은 확인 단계를 제공한다.

- 서버 속도
  - 상품 목록은 페이지 단위로 조회한다.
  - 이미지 파일은 크기 제한을 둔다.
  - 검색 요청은 과도한 부하가 발생하지 않도록 제한한다.
  - 주요 API는 일반적인 환경에서 1~2초 이내 응답을 목표로 한다.

- 유지보수
  - 인증, 상품, 채팅, 지갑, 신고, 관리자 기능을 모듈별로 분리한다.
  - 공통 입력값 검증과 권한 검증 로직을 재사용한다.
  - 테스트가 가능한 구조로 API를 작성한다.
