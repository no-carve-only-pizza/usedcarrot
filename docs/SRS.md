# UsedCarrot 소프트웨어 요구사항 명세서(SRS)

## 1. 문서 개요

### 1.1 목적

본 문서는 중고거래 플랫폼 `UsedCarrot`의 소프트웨어 요구사항을 정의한다. 개발 전 단계에서 시스템의 기능, 보안, 품질, 데이터, 제약사항을 명확히 하여 설계, 구현, 테스트의 기준 문서로 사용한다.

본 프로젝트는 시큐어코딩 수업 과제로, 일반적인 중고거래 서비스 기능뿐 아니라 인증, 인가, 입력값 검증, 개인정보 보호, 안전한 파일 처리, 로그 및 감사 추적 등 보안 요구사항을 핵심 요구사항으로 포함한다.

### 1.2 범위

UsedCarrot은 사용자가 중고 물품을 등록하고, 다른 사용자와 채팅 또는 문의를 통해 거래를 진행할 수 있는 웹 기반 중고거래 플랫폼이다.

주요 범위는 다음과 같다.

- 회원가입, 로그인, 로그아웃
- 사용자 프로필 관리
- 중고 물품 게시글 등록, 조회, 수정, 삭제
- 상품 검색 및 카테고리 필터링
- 관심 상품 등록
- 거래 문의 또는 채팅
- CarrotCoin 모의지갑을 통한 사용자 간 거래
- 거래 상태 관리
- 신고 기능
- 관리자용 사용자, 게시글, 신고 관리
- 보안 로그 및 기본 감사 추적

### 1.3 용어 정의

| 용어 | 설명 |
| --- | --- |
| 사용자 | UsedCarrot에 가입하여 상품을 등록하거나 거래에 참여하는 회원 |
| 판매자 | 상품 게시글을 등록한 사용자 |
| 구매자 | 상품 구매 의사를 가진 사용자 |
| 관리자 | 서비스 운영 및 제재 권한을 가진 사용자 |
| 상품 게시글 | 판매자가 등록한 중고 물품 정보 |
| 거래 상태 | 판매중, 예약중, 거래완료 등의 상품 거래 진행 상태 |
| CarrotCoin | UsedCarrot 내부에서만 사용하는 시큐어코딩 학습용 모의 가상화폐 포인트 |
| 모의지갑 | 실제 블록체인이나 외부 결제망과 연결되지 않고 DB 내부에서만 CarrotCoin 잔액을 관리하는 지갑 |

## 2. 전체 설명

### 2.1 제품 관점

UsedCarrot은 웹 브라우저에서 사용하는 독립적인 중고거래 플랫폼이다. 일반 사용자는 상품을 등록하고 검색하며, 구매 의사 표현 또는 채팅을 통해 거래를 진행한다. 관리자는 서비스 내 신고, 부적절한 게시글, 악성 사용자를 관리한다.

### 2.2 사용자 유형

| 사용자 유형 | 주요 권한 |
| --- | --- |
| 비회원 | 상품 목록 조회, 상품 상세 조회, 검색 |
| 회원 | 비회원 기능, 상품 등록, 관심 상품 등록, 채팅, 신고, 프로필 수정 |
| 판매자 | 본인 상품 수정, 삭제, 거래 상태 변경 |
| 관리자 | 사용자 관리, 게시글 숨김/삭제, 신고 처리, 제재 관리 |

### 2.3 운영 환경

- 클라이언트: 최신 Chrome, Edge, Safari, Firefox
- 서버: Linux 기반 서버 또는 로컬 개발 환경
- 데이터베이스: 관계형 데이터베이스 또는 문서형 데이터베이스
- 네트워크: HTTPS 적용을 전제로 설계

### 2.4 주요 제약사항

- 사용자의 비밀번호는 평문으로 저장하지 않는다.
- 개인정보는 서비스 제공에 필요한 최소 범위만 수집한다.
- 사용자 입력값은 서버 측에서 반드시 검증한다.
- 파일 업로드는 이미지 파일로 제한하며 확장자, MIME 타입, 크기를 검증한다.
- 관리자 기능은 일반 사용자와 명확히 분리한다.
- 보안 관련 이벤트는 추적 가능한 로그로 남긴다.

### 2.5 가정 및 의존성

- 사용자는 이메일 또는 아이디 기반 계정을 사용한다.
- CarrotCoin 기능은 실제 가상자산 네트워크, 개인키, 외부 결제 API와 연동하지 않고 DB 기반 모의지갑으로 구현한다.
- 실시간 채팅은 프로젝트 범위와 일정에 따라 1:1 문의 게시판 또는 간단한 메시지 기능으로 대체할 수 있다.
- 배포 환경에서는 HTTPS가 제공된다고 가정한다.

## 3. 기능 요구사항

### 3.1 회원 관리

| ID | 요구사항 |
| --- | --- |
| FR-AUTH-001 | 사용자는 이메일, 비밀번호, 닉네임을 입력하여 회원가입할 수 있어야 한다. |
| FR-AUTH-002 | 시스템은 중복 이메일과 중복 닉네임을 허용하지 않아야 한다. |
| FR-AUTH-003 | 사용자는 이메일과 비밀번호로 로그인할 수 있어야 한다. |
| FR-AUTH-004 | 로그인한 사용자는 로그아웃할 수 있어야 한다. |

| --- | --- |

| FR-POST-004 | 사용자는 상품 목록을 최신순으로 조회할 수 있어야 한다. |
| FR-POST-005 | 사용자는 상품 상세 정보를 조회할 수 있어야 한다. |

| FR-PAY-006 | 판매자는 본인 상품 판매로 발생한 CarrotCoin 입금 내역을 조회할 수 있어야 한다. |

| FR-PAY-008 | 구매 완료된 상품은 거래완료 상태로 변경되어야 한다. |

## 4. 보안 요구사항

### 4.1 인증

| ID | 요구사항 |
| --- | --- |
| SR-AUTH-001 | 비밀번호는 안전한 단방향 해시 알고리즘과 솔트를 사용하여 저장해야 한다. |
| SR-AUTH-002 | 로그인 실패 시 구체적인 실패 원인을 노출하지 않아야 한다. |
| SR-AUTH-003 | 일정 횟수 이상 로그인 실패 시 추가 인증 또는 일시 제한을 적용할 수 있어야 한다. |
| SR-AUTH-004 | 인증 토큰 또는 세션 ID는 충분히 예측 불가능해야 한다. |
| SR-AUTH-005 | 로그아웃 시 서버 측 세션 또는 토큰 상태를 무효화해야 한다. |

### 4.2 인가

| ID | 요구사항 |
| --- | --- |
| SR-AUTHZ-001 | 사용자는 본인 리소스에 대해서만 수정, 삭제 요청을 수행할 수 있어야 한다. |
| SR-AUTHZ-002 | 관리자 API는 관리자 권한을 가진 사용자만 접근할 수 있어야 한다. |
| SR-AUTHZ-003 | 서버는 클라이언트에서 전달한 사용자 ID를 신뢰하지 않고 인증 컨텍스트를 기준으로 권한을 판단해야 한다. |
| SR-AUTHZ-004 | 메시지, 관심 상품, 신고 내역 등 비공개 데이터는 소유자 또는 관리자만 조회할 수 있어야 한다. |

### 4.3 입력값 검증 및 출력 인코딩

| ID | 요구사항 |
| --- | --- |
| SR-INPUT-001 | 모든 사용자 입력값은 서버 측에서 타입, 길이, 필수 여부, 허용 문자 기준으로 검증해야 한다. |
| SR-INPUT-002 | 상품명, 설명, 닉네임, 메시지 내용에 스크립트가 저장되어 실행되지 않도록 처리해야 한다. |

| SR-INPUT-005 | API 오류 응답에 스택 트레이스, DB 쿼리, 내부 경로를 노출하지 않아야 한다. |

- 하나의 상품은 반드시 한 명의 판매자에게 속해야 한다.
- 관심 상품은 사용자와 상품의 조합이 중복되지 않아야 한다.
- 삭제되거나 숨김 처리된 상품은 일반 사용자에게 노출되지 않아야 한다.

- 관리자 권한은 일반 회원가입 과정에서 직접 선택할 수 없어야 한다.
- 하나의 CarrotCoin 거래는 반드시 하나의 상품, 한 명의 구매자, 한 명의 판매자와 연결되어야 한다.
- 완료된 CarrotCoin 거래 상태는 일반 사용자가 취소 또는 실패 상태로 임의 변경할 수 없어야 한다.
- 동일한 멱등성 키를 가진 CarrotCoin 구매 요청은 중복 처리되지 않아야 한다.
- 지갑 잔액은 0보다 작아질 수 없어야 한다.

## 9. 수용 기준

### 9.1 기능 수용 기준

- 회원은 가입, 로그인, 로그아웃을 정상 수행할 수 있다.
- 회원은 상품을 등록, 조회, 수정, 삭제할 수 있다.
- 판매자가 아닌 사용자는 타인의 상품을 수정하거나 삭제할 수 없다.
- 사용자는 키워드와 카테고리로 상품을 검색할 수 있다.
- 회원은 관심 상품을 등록하고 취소할 수 있다.
- 회원은 상품 판매자에게 메시지를 보낼 수 있다.
- 회원은 거래 상품을 CarrotCoin으로 구매하고 본인 지갑 잔액과 거래 내역을 조회할 수 있다.
- 관리자는 신고 목록을 조회하고 처리할 수 있다.

### 9.2 보안 수용 기준

- 비밀번호는 데이터베이스에 평문으로 저장되지 않는다.
- 서버 측 권한 검증 없이 리소스 변경이 발생하지 않는다.
- 스크립트가 포함된 상품명, 설명, 메시지를 입력해도 브라우저에서 실행되지 않는다.
- SQL 인젝션성 입력값으로 인증 우회나 데이터 조회 조작이 발생하지 않는다.
- 허용되지 않은 파일 업로드는 거부된다.
- 관리자 API는 일반 회원 계정으로 호출할 수 없다.
- 타인의 지갑 잔액이나 CarrotCoin 거래 내역을 조회할 수 없다.
- 동일 CarrotCoin 구매 요청이 반복 제출되어도 이중 처리되지 않는다.
- CarrotCoin 거래 후 구매자와 판매자의 지갑 잔액이 정확히 반영된다.
- 로그인 실패, 권한 오류, 관리자 조치가 보안 로그에 기록된다.

## 11. 개발 범위 우선순위

### 11.1 1차 구현 범위

- 회원가입, 로그인, 로그아웃
- 상품 등록, 목록, 상세, 수정, 삭제
- 서버 측 입력값 검증
- 비밀번호 해시
- 기본 권한 검증
- 이미지 업로드 제한

### 11.2 2차 구현 범위

- 검색 및 필터
- 관심 상품
- 메시지 기능
- CarrotCoin 모의지갑 잔액 조회 및 상품 구매
- 신고 기능
- 보안 로그

### 11.3 3차 구현 범위

- 관리자 페이지
- 사용자 정지
- 신고 처리 워크플로
- 로그인 실패 제한
- CarrotCoin 환불 또는 관리자 잔액 조정 고도화
- 보안 테스트 자동화

## 12. 추적성 매트릭스

| 요구사항 영역 | 관련 기능 | 관련 테스트 |
| --- | --- | --- |
| 인증 | FR-AUTH, SR-AUTH | 회원가입, 로그인, 로그아웃, 비밀번호 해시 테스트 |
| 인가 | SR-AUTHZ | 상품 수정/삭제 권한 테스트, 관리자 API 접근 테스트 |
| 상품 | FR-POST | 상품 CRUD 테스트 |
| 검색 | FR-SEARCH | 키워드/카테고리 검색 테스트 |
| 메시지 | FR-MSG | 메시지 전송, 조회 권한 테스트 |
| CarrotCoin 모의지갑 | FR-PAY, SR-PAY | 지갑 생성, 상품 구매, 잔액 변경, 중복 처리, 조회 권한 테스트 |
| 신고 | FR-REPORT, FR-ADMIN | 신고 등록, 관리자 처리 테스트 |
| 입력값 보안 | SR-INPUT | XSS, SQL 인젝션 방어 테스트 |
| 파일 보안 | SR-FILE | 확장자, MIME, 크기 검증 테스트 |
| 로깅 | SR-LOG | 보안 이벤트 로그 기록 테스트 |

## 13. 미정 사항

- 최종 기술 스택
- CarrotCoin 기능을 필수 범위로 구현할지 선택 범위로 둘지 여부
- 실시간 채팅 구현 여부
- 지역 인증 기능 포함 여부
- 배포 환경 및 HTTPS 인증서 적용 방식
- 개인정보 보관 및 삭제 정책의 상세 수준
