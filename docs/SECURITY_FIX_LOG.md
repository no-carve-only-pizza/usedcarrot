# UsedCarrot 보안 약점 발견 및 수정 이력

이 문서는 개발 과정에서 발견한 보안 약점과 수정 내용을 기록하기 위한 문서이다. 최종 보고서의 "개발 과정에서 확인한 보안 약점들과 변경 내용" 섹션에 이 내용을 요약해서 포함한다.

## 기록 방법

각 보안 약점은 아래 형식으로 기록한다.

```text
## YYYY-MM-DD - 발견 항목 제목

- 발견 위치:
- 위험도: 낮음 / 중간 / 높음 / 치명적
- 발견 방법:
- 문제 설명:
- 공격 가능 시나리오:
- 수정 내용:
- 수정한 파일:
- 검증 방법:
- 남은 위험:
```

## 보안 약점 기록

### 예시 1 - 상품 수정 시 소유자 검증 누락

- 발견 위치: ProductController, ProductService
- 위험도: 높음
- 발견 방법: 테스트 중 다른 사용자의 상품 ID로 수정 요청을 보내 확인
- 문제 설명: URL의 productId만으로 상품을 수정하면 다른 사용자의 상품을 수정할 수 있다.
- 공격 가능 시나리오: 공격자가 `/products/{id}/edit`의 id 값을 바꾸어 타인의 상품 정보를 변경한다.
- 수정 내용: ProductService에서 상품의 seller.id와 현재 로그인한 user.id를 비교하고, 다르면 403 예외를 발생시킨다.
- 수정한 파일: ProductService.java, ProductController.java, ProductAuthorizationTest.java
- 검증 방법: 타인 상품 수정 요청 시 403이 반환되는지 MockMvc 테스트와 실제 화면에서 확인한다.
- 남은 위험: 관리자 기능에서 상품 상태 변경 권한이 과도하지 않은지 추가 검토 필요

### 예시 2 - 상품 설명 XSS 가능성

- 발견 위치: products/detail.html
- 위험도: 높음
- 발견 방법: 상품 설명에 `<script>alert(1)</script>` 입력
- 문제 설명: 사용자 입력값을 HTML로 렌더링하면 스크립트가 실행될 수 있다.
- 공격 가능 시나리오: 악성 판매자가 상품 설명에 스크립트를 삽입하여 다른 사용자의 세션 또는 화면을 공격한다.
- 수정 내용: Thymeleaf에서 `th:utext`를 사용하지 않고 `th:text`로 출력한다.
- 수정한 파일: products/detail.html, products/list.html
- 검증 방법: 스크립트 입력 후 상세 페이지에서 코드가 문자로 표시되고 실행되지 않는지 확인한다.
- 남은 위험: 추후 마크다운 또는 HTML 허용 기능을 추가할 경우 sanitizer 필요

### 예시 3 - 파일 업로드 검증 부족

- 발견 위치: FileStorageService
- 위험도: 중간
- 발견 방법: 확장자만 jpg로 바꾼 실행 파일 업로드 시도
- 문제 설명: 확장자만 확인하면 실행 파일이나 비이미지 파일이 업로드될 수 있다.
- 공격 가능 시나리오: 공격자가 악성 파일을 이미지로 위장하여 서버 저장소에 업로드한다.
- 수정 내용: 확장자, MIME 타입, 파일 크기, 파일 개수를 모두 검증하고 저장 파일명은 UUID로 변경한다.
- 수정한 파일: FileStorageService.java, ProductController.java
- 검증 방법: exe 파일, 위장 jpg 파일, 5MB 초과 파일, 5장 초과 업로드가 거부되는지 확인한다.
- 남은 위험: 운영 환경에서는 별도 파일 서버나 오브젝트 스토리지 사용 권장
# UsedCarrot 보안 수정 기록

## 2026-06-28 구현 중 반영

- 비밀번호 평문 저장 위험: `User.passwordHash`와 BCrypt `PasswordEncoder`를 사용하도록 구현했다.
- CarrotCoin 외부 연동 오해 위험: 실제 블록체인/개인키/외부 API 없이 DB 내부 `Wallet`, `WalletTransaction`만 사용하도록 구현했다.
- 금액 변조 위험: 구매 금액을 클라이언트 입력에서 받지 않고 서버에 저장된 상품 가격만 사용한다.
- 중복 구매 위험: `idempotencyKey` unique 제약과 기존 거래 반환 로직을 추가했다.
- 파일 업로드 우회 위험: 확장자, MIME 타입, 파일 크기, 이미지 시그니처를 모두 확인하고 UUID 파일명으로 저장한다.
- 권한 우회 위험: 상품 수정/삭제, 채팅방 조회/전송, 지갑 조회와 거래 내역 조회를 인증 사용자 기준으로 검증한다.
- 민감정보 로그 노출 위험: 감사 로그 상세 내용에서 password/token/session/secret 키워드를 마스킹한다.

## 2026-06-28 보조 에이전트 검토 후 추가 수정

- H2 콘솔 노출 위험: `/h2-console/**` 공개 허용과 CSRF 예외를 제거하고 `spring.h2.console.enabled=false`로 기본 비활성화했다.
- CarrotCoin 동시 구매 이중 판매 위험: 구매 처리 시 상품 행을 `PESSIMISTIC_WRITE`로 잠근 뒤 상태를 검증하도록 수정했다.
- 멱등성 키 오용 위험: 기존 `idempotencyKey` 거래 반환 전 구매자와 상품이 현재 요청과 같은지 확인하도록 수정했다.
- 제한 사용자 권한 우회 위험: `LIMITED` 사용자가 본인 상품을 수정/삭제할 수 없도록 상태 검증을 추가했다.
- 상품 수정 폼 정보 노출 위험: edit form 조회 시에도 판매자 본인 또는 관리자 권한을 검사하도록 수정했다.
- 숨김/삭제 상품 이미지 직접 접근 위험: 정적 `/uploads/**` 파일 매핑을 제거하고 DB의 상품 상태를 확인하는 컨트롤러로 이미지를 제공하도록 수정했다.

## 2026-07-12 모의해킹 후 수정

- 공개 기본 관리자 계정 탈취: 코드에 고정된 관리자/일반 계정을 제거하고 환경변수로 입력한 관리자만 최초 생성하도록 변경했다.
- 다계정 CarrotCoin 인플레이션: 회원가입 지갑을 0 CC로 생성하고 관리자만 사용자별 최대 1,000,000 CC를 지급하도록 변경했다.
- 다계정 허위 신고 제재: 신고 접수 단계의 자동 제재를 제거하고 관리자가 `RESOLVED`로 승인한 신고만 누적하도록 변경했다.
- 제3자 계정 잠금 서비스 거부: 사용자별 5회 잠금을 제거하고 IP별 60초 10회 로그인 실패 제한으로 변경했다.
- 감사 로그 IP 위조: 임의 `X-Forwarded-For`를 무시하고 설정된 신뢰 프록시에서 들어온 경우에만 전달 IP를 사용한다.
- 상품 쓰기 경로 공개: `/products/new`, `/products/me`를 공개 상품 상세 패턴보다 먼저 인증 대상으로 지정했다.
- 오류 상태 은폐: `AppException`을 오류 유형에 따라 HTTP 400, 403, 404, 409로 응답하도록 수정했다.
- 세션/브라우저 방어 부족: 운영 기본 쿠키에 `Secure`, `HttpOnly`, `SameSite=Strict`를 적용하고 CSP와 프레임 차단 헤더를 추가했다.
- 이미지 위장 업로드: 확장자와 MIME 타입의 일치 여부를 확인하고 JPEG/PNG는 실제 디코딩 및 최대 2천만 픽셀 제한까지 검증하도록 강화했다. WebP는 RIFF/WEBP/VP8 헤더를 추가 확인한다.
- 검증: 기본 계정 미생성, 0 CC 지갑, 관리자 지급, 승인형 신고, IP 제한, 프록시 헤더 무시, 인증 경로 및 404 상태를 자동 테스트로 추가했다.
